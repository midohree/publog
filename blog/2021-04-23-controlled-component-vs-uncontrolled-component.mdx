## 서론

기본적인 회원가입과 로그인 부터 인터넷으로 물건을 주문할 때, 구글에 정보 검색을 할 때,
채팅으로 메세지를 전송할 때 처럼 사용자로부터 데이터를 입력 받기 위해 사용 되는 양식(form)은 웹 애플리케이션에서 필수적인 요소이다.
하지만 리액트로 직접 구현을 해보면, 여간 귀찮은 일이 아닐 수 없을 정도로 고려해야 할 부분이 많다는 것을 느끼게 된다.
아래는 닉네임을 입력하는 기본적인 컴포넌트이다.

```jsx live
function LogIn() {
  const [nickname, setNickname] = useState('');

  function handleChange(e) {
    const { value } = e.target;
    setNickname(value);
  }

  function handleSubmit(e) {
    e.preventDefault();
    alert(`nickname: ${nickname}`);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type='text'
        name='nickname'
        onChange={handleChange}
        value={nickname}
      />
      <input type='submit' value='제출' />
    </form>
  )
}
```

## Controlled Component

위 예제에 사용된 인풋들은 **Controlled Component(제어 컴포넌트)**이다.

<Note title='Controlled Component'>
  onChange와 같은 콜백을 사용하여 다른 컴포넌트에 props를 통해 변경 사항을 새로운 값으로 전달하는 컴포넌트.
</Note>

위 예제에서 제어 컴포넌트가 작동하는 방식을 하나씩 기술해보자면 아래와 같다.

1. 처음 nickname의 상태는 빈 문자열이다. `(' ')`
2. `ㄱ` 을 입력하면 handleChange 함수가 `ㄱ` 을 가져와 상태로 저장하고 인풋에 `ㄱ`을 새로운 값으로 전달한다. (push)
3. 인풋은 `ㄱ`을 값으로 가지기 위해 페이지를 리렌더링하고 onChange 함수를 다시 생성한다.
4. `ㅏ` 를 입력하면 handleChange 함수가 `가` 를 가져와 상태로 저장하고 인풋에 `가`를 **새로운 값으로 전달한다. (push)**
5. 인풋은 `가`를 값으로 가지기 위해 페이지를 리렌더링하고 onChange 함수를 다시 생성한다.

결론적으로 얘기하자면 입력값이 변화할 때 마다 값을 상태(state)로 저장해야 하고, 저장된 상태를 새로운 값으로 인풋에 전달(prop)하기 때문에 리렌더링이 발생한다. 제어 컴포넌트는 이런식으로 변경 사항을 새로운 값으로 전달해 UI를 업데이트 시키는데, 값이 변경 될 때 마다 불필요한 렌더링이 발생할 뿐더러 컴포넌트 안에 선언된 함수들도 계속 해서 새로 생성시키기 때문에 비효율적이다. 이 문제를 해결하기 위해서 `useCallback`이나 `useMemo`를 통해 최적화 시켜줄 수 있다.

```jsx
const handleChange = useCallback((e) => {
  const { value } = e.target;
  setNickname(value);
}, []);
```

그럼에도 불구하고 제어 컴포넌트로 작업하는 것은 약간 성가시다. 만약 입력폼 개수가 많아진다면 각각의 입력폼에 `value` 속성과 `onChange` 핸들러를 작성해줘야 한다. 폼의 목적에 대해 생각해 봤을 때도 값이 변경 될 때 마다 어떤 로직을 수행시키는 것이 아니라면, 폼이 결국 제출 되고 난 후의 데이터가 필요한거지 각각의 입력값이 변화되는 상태를 알고싶은건 아니라 변화값을 상태로 저장하는건 어쩐지 이상하게 느껴진다. 이런 상황에 비제어 컴포넌트가 필요해진다.

## Uncontrolled Component

상태정의와 이벤트 핸들러를 작성하지 않고, 페이지가 리렌더링 되는 것을 방지 하기 위해 **Uncontrolled Component(비제어 컴포넌트)**를 사용할 수 있다.

<Note title='Uncontrolled Component'>
  각 인풋 요소의 데이터는 컴포넌트가 아닌 DOM에 저장된다. 상태변화를 위해 이벤트 핸들러를 작성 할 필요 없이 ref를 통해 DOM에 접근해 이벤트를 핸들링한다.
</Note>

위의 제어 컴포넌트를 비제어 컴포넌트로 바꿔보았다.

```jsx live
function LogIn() {
  const inputRef = useRef(null);

  function handleSubmit(e) {
    e.preventDefault();
    alert(`nickname: ${inputRef.current.value}`);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input type='text' name='nickname' ref={inputRef} />
      <input type='submit' value='제출' />
    </form>
  )
}
```

그럼 비제어 컴포넌트는 어떻게 동작할까? 간단하다.

1. ref를 이용해 DOM에 접근한다.
2. **필요할 때 인풋의 값을 가져온다. (pull)**

정리하자면,

<Note>
  제어 컴포넌트가 변경값을 새로운 값으로 push 한다면 비제어 컴포넌트는 필요할 때 값을 pull 해온다.
</Note>

컴포넌트 내부에 `console.log('hello world')` 를 입력한 후 두 상황을 비교해보면, 제어 컴포넌트는 입력값이 바뀔 때 마다 출력되는 반면 uerRef는 초기 렌더링에만 출력 되고있다.

## When to use Which ?

확실히 제어 컴포넌트일때 보다 컴포넌트 내부가 깔끔해 보인다. 성능 또한 제어 컴포넌트를 사용했을 때 보다 좋다. `forwardRef` 를 사용하면 하위 컴포넌트에 `ref`를 명시적으로 전달하기도 가능하다. 이렇게만 두고 보면 폼을 다룰 때는 무조건! 비제어 컴포넌트를 사용 해야겠다는 생각이 든다. 하지만 리액트 공식 문서에는 아래와 같이 써있다.

<Note>
  대부분 경우에 폼을 구현하는데 제어 컴포넌트를 사용하는 것이 좋습니다. 제어 컴포넌트에서 폼 데이터는 React 컴포넌트에서 다루어집니다. 대안인 비제어 컴포넌트는 DOM 자체에서 폼 데이터가 다루어집니다.
</Note>

“대부분 경우” 이 단어가 나에게 큰 혼란을 줬다. 그럼 어떤 경우에 제어 컴포넌트, 비제어 컴포넌트를 사용해야 하는걸까?

- onChange 시점의 유효성 검사 결과에 따라 특정 로직을 수행해야 하는 경우
- 많은 form이 서로의 값에 의존적인 경우
- 모든 필드에 유효한 데이터가 없으면 버튼 비활성화
- 신용 카드 번호와 같은 특정 입력 형식 적용
- submit 기능만 적용하면 되는 일회성 입력폼일 경우 (채팅, 댓글)
- 제출 시 검증이 필요한 경우

위에서 말했듯이 “대부분의 경우” 제어 컴포넌트를 사용하지만, 일회성 폼이나 입력값의 변화를 감지하지 않아도 될 상황이라면 비제어 컴포넌트를 사용한다. 그럼 publ에서 사용 되는 인풋은 어떤 방식으로 작성되어야 할지 정리해보았다. 우선 폼이 사용 되는 곳은 아래와 같다.

1. 유효성검사가 가능한 로그인, 회원가입 입력폼
2. 댓글 달기 입력폼
3. 채팅 입력폼
4. 라이브채팅 입력폼

로그인, 회원가입엔 유효성 검사가 필수이다.
유효성 검사는 데이터가 입력 될 때마다, 입력 후 인풋에 포커스 아웃이 되었을 때, 혹은 모든 데이터를 입력 후 제출 버튼을 눌렀을 때 실행 시킬 수 있고,
어떤 방식이 사용자 경험에서 더 편리하고 알아보기 쉬울지 기획 의도에 맞춰 적용 시킬 수 있다.

만약 사용자가 데이터를 입력할 때 마다 검증을 하고 싶다면 onChange 될 때 마다 검증 로직을 실행시켜야 하기 때문에 제어 컴포넌트를,
제출 후 어떤 필드에서 에러가 났는지 확인하고 싶다면 인풋의 값이 변화하는지 알 필요가 없기 때문에 비제어 컴포넌트를 사용하는게 좋다.
개인적으로 사용자 입장일 때 로그인과 회원가입 시 불필요한 시간 낭비 하는걸 좋아하지 않기 때문에 실시간으로 검증 상태를 알 수 있는 onChage로 작업했다.
더불어 모든 입력폼의 데이터가 입력되었을 때만 버튼이 활성화 되어야 하기 때문에 제어 컴포넌트를 사용하는게 맞지만, 만약 기획이 변경된다면 비제어 컴포넌트로 바꿔야 하는 상황이 올 수도 있다.

댓글 달기, 채팅, 라이브채팅 같은 경우는 딱히 검증 작업이 필요하지 않다. 제출 되었을 때 인풋에 있는 값이 뭔지만 알면 된다.
따라서 비제어 컴포넌트를 사용하는게 맞는 것 같아보인다.
하지만 인풋에 값이 없을 땐 전송 버튼을 disabled 시켜야하고, 추후에 최대 글자 수 제한 기능이 추가 될 수도 있기 때문에 비제어 컴포넌트를 사용하는게 과연 맞는건지 의문이 든다.

**비제어 컴포넌트로 사용하되, 입력값 유무나 데이터 변화를 알 수 있는 방법이 있다면 아주 좋을 것 같다.**

## React-hook-form

React-hook-form은 폼을 비제어 방식으로 다룰 수 있도록 지원하는 라이브러리이다. 물론 제어 방식도 지원한다.
하지만 React-hook-form 의 중요 컨셉 중 하나는 `register`를 통해 비제어 컴포넌트를 Hook과 연결하여 값이 검사될 수 있도록 만들고 폼을 제출할 때 한꺼번에 모아지도록 하는 것이다.
기존에 사용하던 Formik은 비제어 방식을 제공하고 있지 않기 때문에 React-hook-form으로 변경하기로 했다.

더불어 기본적인 유효성 검사 방식(required, min, max, minLength, maxLength, pattern, validate)을 제공하면서,
폼의 모드를 onChange, onBlur, onSubmit 중에 선택 할 수 있기 때문에 비제어 컴포넌트에서 하기 힘든 `입력값이 변화 될 때 마다 유효성 검사를 실행` 시킬 수 있다는 것도 큰 장점으로 다가왔다.

`useForm()` 을 통해 가져온 메소드들 중에 입력폼의 데이터가 무엇인지 실시간으로 확인 할 수 있는 `watch`와
폼 상태에 대한 정보를 확인 할 수 있는 `formState`를 잘 조합하면 제어 컴포넌트 처럼 사용 할 수 있다.

만약 폼 내부가 깊게 중첩되어 있는 형태라면 prop으로 폼 컨텍스트를 내려주지 않아도, 폼을 `FormProvider`로 감싸준 후
하위 컴포넌트에서  `useFormContext()` 를 이용해 폼 상태를 즉각적으로 불러와 읽거나 사용할 수 있다.

<Note>
  로그인, 회원가입은 유효성 검사 기획이 변경된다면 아래 설명된 코드와 상이할 수 있으나 초반에 어떻게 작성했는지에 대한 참고용으로 봐주세요.
</Note>

### Basic Usage

현재 폼이 필요한 최상단 컴포넌트들의 기본적인 구조는 아래와 같다.

```jsx {4}
import { useForm, FormProvider } from 'react-hook-form';

function SignInForm() {
  const methods = useForm({ mode: 'onChange' });

  const {
    handleSubmit,
    formState: { isValid, touched }
  } = methods; // 필요한 기능들을 불러온다.

  function handleSignIn(data) {
    console.log(data); // form 안에 있는 모든 인풋값이 객체형태로 들어온다.
  }

  return (
    <FormProvider {...methods}>
      <Form onSubmit={handleSubmit(handleSignIn)}>
        <InputFieldButton name='email' cancel />
        <InputFieldButton type='password' name='password' cancel />

        <CheckBox marginTop='16' name='service' />

        <SubmitButton full />
      </Form>
    </FormProvider>
  );
}
```

일단 폼의 모드를 `onChange`로 설정했다. 디폴트 값은 onSubmit이다.
그 후 `FormProvider`를 통해 하위 모든 메소드들을 컨텍스트 내부로 전달한다. 그럼 `register`를 사용할 제일 최하단에 네스팅 되어있는 인풋 컴포넌트를 보자.

```jsx
function TextFieldButton({ ...rest }) {
  const { register } = useFormContext();

  // code...

  return (
    <Input ref={register} {...rest} />
  );
}
```

`register`를 props로 내려주는 방식이 아닌 `useFormContext()` 를 통해 불러온다.
`ref` 속성에 register를 부여하면 **react-hook-form이 알아서 해당 인풋 정보를 가져온다.**

UI로 사용되는 가장 작은 단위의 인풋과 버튼에는 이처럼 `useFormContext()` 로 필요한 메소드들을 불러 와서 사용하고 있기 때문에
최상단 컴포넌트에서 내가 이 폼을 어떻게 사용할 것인지만 정해주면 된다.

### Validation

react-hook-form에서 기본적인 유효성 검사기능을 제공한다. 물론 커스텀해서 사용 할 수도 있다.
단 유효성 검사를 위해서 `name` 속성은 필수이며 유니크한 값으로 설정되어야 한다. 기본적인 사용법은 아래와 같다.

```jsx
<input name='test' ref={register({ required: true, minLength: 10 })} />
```

폼 제출을 위해 반드시 필요한 경우 `required: true` 로 설정해주고 그 외에도 유효성 검사에 필요한 프로퍼티와 값을 설정해주면 조건에 맞게 유효성 검사를 실행한다.
각 검사에 해당되는 에러 메세지 또한 지정해서 사용 할 수 있다.

```jsx
<input
  name='test'
  ref={register({
    required: {
      message: 'This is Required'
    },
    minLength: {
      value: 10,
      message: 'Too short Characters'
    }
  })}
/>
```

메세지를 설정하면 `errors` 객체를 통해 에러 타입과 메세지를 가져 올 수 있다.
제공해주는 유효성 검사 기능 외에도 원하는 방식으로 커스텀 할 수도 있다.

```jsx
<input
  name='num'
  placeholder='must be a number between 0 and 10 that is not 5'
  ref={register({
    validate: {
      positive: (value) => parseInt(value, 10) > 0,
      lessThanTen: (value) => parseInt(value, 10) < 10,
      notFive: (value) => parseInt(value, 10) !== 5
    }
  })}
/>

{errors.num?.type === 'positive' && <p>Only positive numbers accepted</p>}
{errors.num?.type === 'lessThanTen' && <p>Only numbers less then 10</p>}
{errors.num?.type === 'notFive' && <p>The number 5 is not accepted</p>}
```

Formik으로 작성된 유효성 검사 부분이 yup을 사용했기에 나도 yup으로 유효성 검사 로직을 작성하기로 했다.
심지어 react-hook-form에서 공식적으로 `Yup`의 표준 리졸버를 지원하고 있다.
하지만 추가 패키지를 인스톨 하기 이전에 일단은 커스텀 리졸버를 작성해 훅처럼 사용하기로 했다.

```jsx
const useYupValidationResolver = (validationSchema) =>
  useCallback(
    async (data) => {
      try {
        const values = await validationSchema.validate(data, {
          abortEarly: false
        });

        return {
          values,
          errors: {}
        };
      } catch (errors) {
        return {
          values: {},
          errors: errors.inner.reduce(
            (allErrors, currentError) => ({
              ...allErrors,
              [currentError.path]: {
                type: currentError.type ?? 'validation',
                message: currentError.message
              }
            }),
            {}
          )
        };
      }
    },
    [validationSchema]
  );
```

데이터 인자를 받아서 `validate` 한 후 에러가 없으면 values와 빈 객체를 반환시킨다. (기본적으로 errors는 빈 객체이다.)
만약 에러가 발생했을 때는 errors 객체의 키 값을 인풋과 연결지어줘서 어떤 인풋에서 에러가 났는지 명시해주는 객체를 반환헤준다.
일단 커스텀 리졸버로 작성하긴 했지만 제공해주는 표준 리졸버로 추후 리팩토링 할 지도 고려중이다.

### Issue #1 - 깜빡거리는 Submit 버튼

로그인과 회원가입 페이지에서 처음엔 제출버튼이 비활성화 되어있는 상태이고, 인풋의 유효성 검사가 “모두” 통과 해야만 버튼을 활성화 시키게끔 했다.
formState의 `isValid`는 에러가 없다면, (즉 모든 인풋이 유효성 검사를 통과했다면) true를 반환한다.
그래서 버튼의 disabled 속성에 `isValid` 값을 넣어 유효성 검사를 모두 통과 하기 전까지는 비활성화 되게끔 작성했다.

```jsx {8}
import { useForm, FormProvider } from 'react-hook-form';

function SignInForm({ onSignIn }) {
  const methods = useForm({ mode: 'onChange' });

  const {
    handleSubmit,
    formState: { isValid }
  } = methods; // 필요한 기능들을 불러온다.

  return (
    <FormProvider {...methods}>
      <Form onSubmit={handleSubmit(onSignIn)}>
        <InputFieldButton name='email' cancel />
        <InputFieldButton type='password' name='password' cancel />

        <CheckBox marginTop='16' name='service' />

        <SubmitButton full disabled={!isValid}/> //-----> 이 부분
      </Form>
    </FormProvider>
  );
}
```

그런데 로그인과 회원가입 페이지를 토글했을 때 최초 페이지 렌더링에 버튼이 0.1초 정도 활성화가 되는 현상이 발생했다.
깜빡! 하고 활성화가 되었다가 다시 비활성화로 돌아온다.

찰나의 순간이지만 분명히 거슬린다. 분명히 잠시 버튼이 활성화된다.
처음엔 버튼의 disabled 속성이 어딘가에서 다른 값을 받고 있는건지, 아니면 내가 로직을 잘못 작성한건지 찾아봤지만 다른 곳에서 문제가 있진 않았다.
`isValid` 를 콘솔에 찍어보니 답이 나왔다.

초기 마운트 시 false 값이여야 할 isValid가 true로 나온다. (완전 배신감..)
깃헙 이슈를 뒤져보니 나와 비슷한 상황의 사람이 있었고, 왜 isValid의 기본 값이 true인지에 대한 이유가 나온다.

초기값이 false라면 ?

  - input 1의 유효성 검사를 통과 했을 때 👉  true를 렌더한다.
  - input 2의 유효성 검사를 실패 했을 때 👉  false를 렌더한다.

초기값이 true라면 ?

  - input 1의 유효성 검사를 통과 했을 때 👉  그대로 true이다.
  - input 2의 유효성 검사를 실패 했을 때 👉  false를 렌더한다.

초기값이 false라면 폼이 유효하던 말던 여러번의 리렌더링이 발생하게 될거고 그 이유 때문에 리렌더링을 최소한으로 유지 하기 위해 true로 설정했다는 것이다.
(참고 링크 :  isValid return true in initial render)

그래서 추가적으로 초기 렌더링 시 버튼을 확!실! 하게 비활성화 시킬 방법이 필요했고, formState의 다른 메서드인 `touched`를 같이 사용해보기로 했다.
touched는 초기에는 빈 객체이고 인풋에 값이 찍히면 true를 값으로 하는 프로퍼티가 생성된다. `e.g. { email: true }`

```jsx
  const {
    handleSubmit,
    formState: { isValid, touched }
  } = methods;

  const isAllTouched = Object.values(touched).length > 0;
  const isButtonActive = isValid && isAllTouched;

  // code...

  <SubmitButton full disabled={!isButtonActive} />
```

모든 인풋에 값이 있으면서도 유효성 검사를 통과해야지 버튼을 활성화 시키는 조건을 달아줬다.
반대로 얘기하자면 인풋이 건드려지지 않음과 유효성 검사를 통과하지 않았다면, 즉 아무런 짓도 하지 않았다면 버튼을 비활성화한다.
이렇게 해 두니 버튼이 깜빡거리는 현상은 방지되었다.
